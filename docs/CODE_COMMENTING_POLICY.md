# Code Commenting Policy

## Why not “comment every line”

Тотальное комментирование каждой строки быстро устаревает и усложняет поддержку. В проекте используется практичный стандарт: документировать intent, контракты и неочевидные решения.

## Mandatory Comments

- Заголовок модуля (что делает файл).
- Public функции/экспорты (input/output/side effects).
- Сложные вычисления, где неочевиден мотив.
- Нетипичные бизнес-правила и ограничения.

## Avoid

- Комментарии уровня “присвоили переменной значение”.
- Дублирование названий переменных в тексте.
- Длинные комментарии, не привязанные к логике.

## Recommended Format

- Для JS/TS: JSDoc на экспортируемые функции.
- Для страниц React: короткий блок о состоянии и побочных эффектах.
- Для API: ссылка на `docs/API.md`.

## Definition of Done for Documentation

- Есть актуальный docs-каталог.
- У ключевых модулей понятна ответственность.
- Публичные интерфейсы описаны.
- Из README можно перейти к нужной документации за 1 шаг.
